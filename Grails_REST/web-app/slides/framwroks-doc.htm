JavascriptMVC

Introducción

JavaScriptMVC es un framework para el desarrollo de aplicaciones JavaScript, y se presenta como una extensión de la librería jQuery. Pretende complementar esta librería para permitir el desarrollo de aplicaciones JavaScript de calidad empresarial.

A la vez, este framework basa su comunicación con los servidores en los principios REST, aunque esto no limita el uso del framwork bajo, la comunicación con los servidores se vuelve mucho mas sencilla y natural cuando cumplimos estos principios.

Por tanto, el presente documento introduce tanto los conceptos básicos de jQuery como los principios que establece REST. Para terminar haciendo una introducción la framework JavaScriptMVC.

El objetivo final no es tanto el de explicar como usar el framework, sino más bien entender que aporta y que soluciona este, y por tanto en caso de presentarse un proyecto que cumpla ciertos criterios y envergadura, poder tomar la decisión de si JavaScriptMVC presenta las herramientas necesarias para el desarrollo de nuestro proyecto.

Representational State Transfer

Introducción a REST

En los años 90, en pleno auge de Internet y la WWW, un grupo de investigadores fue encargado de mejorar la arquitectura en la que se basaba la WWW de tal forma que soportara las previsiones de crecimiento exponencial que ya por aquella época se intuían. Para ello, un integrante del grupo, Roy T. Fielding realizó un análisis comparando distintos estilos arquitectónicos en busca de aquel que mejor cubriese los requisitos de la Web y aportara el mayor número de propiedades arquitectónicas.

Como fruto de esta investigación obtuvo la siguiente tabla comparativa, que es una comparación de estilos arquitectónicos software para el diseño de una aplicación de Hypermedia basada en la Web en función de las propiedades que aporta:

El estilo que más propiedades aportaba fue Layered-Code-on-Demand-Client-Cache-Stateless-Server (LCODC$SS) cuyas características son:

  1. Layered: Estilo multicapa en el que existen cuatro componentes (Cliente, Proxy, Gateway, Servidor). Los clientes hacen peticiones, los proxys agrupan peticiones de distintos clientes, los gateway reciben peticiones para un conjunto de servidores y se las retransmiten a estos.
  2. Code-on-Demand: Se basa en que el cliente hace una petición y el servidor le envía código para que el cliente lo interprete. Alivia la carga del servidor haciendo que el cliente realice parte del procesamiento, a cambio de que ambos tengan que estar de acuerdo en las tecnologías a utilizar.
  3. Client-Stateless-Server: Arquitectura cliente servidor en la cual el servidor no guarda ninguna información entre llamadas por lo que el cliente siempre tiene que enviar toda la información requerida por el servidor.
  4. Cache: Los componentes de la arquitectura guardan información de las peticiones permitiendo, en el mejor de los casos, tener que volver a retransmitir al servidor una misma petición.  

Posteriormente Roy T. Fielding propone REST como estilo arquitectónico en el que debía basarse la “Web Moderna”. REST se basa en LCODC$SS con un añadido fundamental, el uso de una Interfaz Uniforme entre componentes. La Interfaz Uniforme consiste en el uso de un formato estándar para la comunicación entre componentes, lo cuál simplifica la complejidad total del sistema permitiendo un desarrollo desacoplado del cliente/servidor e independiente de los servicios que aporten.

Los componentes de la arquitectura REST deben de comunicarse transfiriendo una representación de un recurso en un formato estandar (en el caso de la Web: HTML, XML, JPEG, etc.), seleccionado dinámicamente en función de las necesidades del receptor y de la naturaleza del recurso.

RESTful HTTP

REST es un estilo arquitectónico para sistemas en Red basados en Hypermedia (Documentos distribuidos que enlazan a otros documentos). Y como tal, sólo especifica los beneficios y propiedades de usar este tipo de Arquitectura en el campo de los sistemas en Red basados en Hypermedia, sin entrar en detalle de como realizar una implementación concreta de la Arquitectura.

Debido a lo anterior, se denomina como RESTful a la implementación que cumple con las pautas marcadas por el estilo REST.

El ejemplo más claro de arquitectura RESTful es RESTful HTTP, modelo en el que se basa la Web. RESTful HTTP modela el concepto de Interfaz Uniforme especificado por REST mediante el uso de un “lenguaje de comunicación” basado en el patrón Verbo + Sustantivo:

  1. Verbos: Son definidos por el protocolo HTTP. Existen 6 tipos: GET, POST, PUT, DELETE, HEAD y OPTIONS, aunque en la práctica los servidores web sólo suelen aceptar el uso de los 2 primeros. Además HTTP deja muy clara la semántica de cada uno de los verbos de tal forma que GET es idempotente, lo cual significa que una llamada get no debe modificar el estado del recurso al que se aplica. Por lo tanto, GET es considerado un método “seguro” y facilita la labor a la cache. Por su parte, POST no es un método ni idempotente ni seguro, y normalmente su utiliza para crear un nuevo recurso.
  2. Sustantivos: Son definidos por el protocolo URI. Identifican de forma inequívoca a recursos dentro de la Web.

Todas las comunicaciones en HTTP siguen esta sencilla estructura.

Una aplicación para la Web que siga el estilo RESTful HTTP debería cumplir los siguientes 5 principios:

  1. Darle a todas las “cosas” un ID: Todos los recursos que deban ser identificables deben poseer una URI propia.
  2. Utilizar links para enlazar las “cosas”: Hay que utilizar links para enlazar recursos siempre que sea posible. El Hyperlinking es lo que hace a la Web la Web.
  3. Usar métodos estándar: Es decir, usar los verbos HTTP con el propósito que fueron definidos.
  4. Recursos con múltiples representaciones: Hay que aceptar distintos formatos a la hora de enviar representaciones de recursos.
  5. Usar comunicaciones sin estado: El servidor no debe guardar ningún tipo de información relativa al cliente entre sucesivas peticiones.

Introducción a Servicios REST

Uno de los principales quebraderos de cabeza a lo largo de la historia del desarrollo de aplicaciones software ha sido la comunicación remota de aplicaciones para la programación distribuida.

Muchos han sido los protocolos diseñados a lo largo de las últimas décadas para dar solución a este problema. Estos protocolos se pueden categorizar en:

  1. Basados en RPC: Su fundamento es la invocación remota de métodos. Habitualmente implica serializar un objeto en un lado de la comunicacion, mandarlo por la red y deserializarlo en el otro extremo. Necesitas una interfaz del objeto en ambos lados de la comunicación. Da lugar a arquitecturas muy complejas. Ejemplos: CORBA, RMI, etc.
  2. Basados en MOM: Se basa en la comunicación asíncrona de componentes mediante la existencia de un “elemento intermedio” que implementa el patrón Productor-Consumidor. Cada vez que un Productor genera un mensaje el elemento intermedio lo capta, y lo pone a servicio de los Consumidores que lo recibirán cuando estén disponibles. Esto favorece el desarrollo independiente de productores y consumidores que no tienen ni siquiera que estar implementados en el mismo lenguaje. Con la definición del estándar JMS, este tipo de arquitecturas han obtenido mucho peso en el desarrollo de aplicaciones que explotan el uso de comunicaciones asíncronas (Ejemplo: Twitter).
  3. Web Services: Surgen a partir de XML-RPC. Sufren un gran auge gracias en parte a su adopción en la arquitectura SOA para el desarrollo de grandes aplicaciones. Extensa literatura en torno a WS-*. Su uso está muy extendido en la web. Su principal defecto es que los protocolos en los que se basa (SOAP, WSDL, UDDI) añaden demasiada complejidad.

Aunque hay muchos modos de implementar la comunicación remota en aplicaciones distribuidas, cada una tiene sus ventajas e inconvenientes, por lo que se pensó en utilizar REST cuyo éxito está demostrado sobradamente en el hecho de que la Web ha sido capaz de sobrevivir durante muchos años sin ningún cambio en su arquitectura.

A diferencia de los servicios web basados en SOAP, no hay ningún estándar oficial que defina como implementar un servicio web basado en REST. Simplemente existe la certeza de que al aplicar el estilo arquitectónico REST obtendremos las mismas propiedades que la Web (la más importante de ella, la simpleza de uso).

Servicios REST en la Web Moderna

La Web considerada actualmente moderna, es aquella en la que es el usuario el que crea el contenido (no un Administrador como antiguamente) y en la que la información generada se expande a través de redes sociales y existen multitud de plataformas de acceso a la información.

En esta Web moderna, aplicaciones como Twitter, Facebook o LinkedIn son los reyes y los servicios REST son la forma de acceder a la información más extendida.

Habitualmente, los contenidos en redes sociales son accedidos a través de aplicaciones externas a la propia red social. Las aplicaciones que agregan contenido de distintas redes sociales están de moda y la forma que tienen de acceder a la información es mediante Servicios REST.

En la Web Moderna los recursos pueden ser:

  1. Públicos: No hay necesidad de utilizar seguridad.
  2. Propiedad del servicio: El cliente debe dar sus credenciales.
  3. Propiedad del usuario: El usuario debe autorizar al cliente.

Los 2 primeros casos están cubiertos por la Arquitectura RESTful HTTP. Para cubrir el tercer caso se diseñó el estándar OAuth.

OAuth (Open Authorization) es un estándar abierto para la autorización que permite al usuario compartir sus recursos privados con otros sitios web.

Actualmente está en desarrollo la versión 2.0 del estándar (existen 22 drafts distintos). La nueva versión es mucho más simple que la anterior y añade distintos tipos de autorización.

La principal característica de OAuth es que el usuario puede ver que permisos está otorgando al sitio externo y en cualquier momento puede revocar el acceso otorgado. Además no tiene que dar su usuario y contraseña al sitio externo, sino que es en el sitio legítimo donde se identificará y podrá aceptar los permisos necesarios para el acceso a sus datos.

Por último, con vistas a facilitar la labor a los desarrolladores encargados de escribir clientes que utilicen una API REST, es fundamental describir formalmente el funcionamiento del API, incluyendo las URLs y el formato de los mensajes. Esto se puede abordar de dos formas:

  1. Diseñando una documentación online entendible por el ser humano.
  2. Utilizando un estándar para formalizar la definición del API REST: Web Application Description Language (WADL) and Web Service Definition Language (WSDL) 2.0. Ambas opciones son difíciles de entender.

Closure

Como vemos, devolvemos “una función (sumar)”, que en otro lenguaje de programación podría ser una referencia a un espacio de memoria donde se encuentra dicha función.

Pero si nos fijamos bien, vemos que utiliza una variable declarada fuera de este “sumar”, con lo que en cualquier otro lenguaje no hubiese funcionado. ¿Que ocurre realmente en jQuery?

Pues ahí está la importancia del closure, es decir lo que ahora mismo es sumar. ¿Que es un closure? Pensemos en una caja en la que dentro hay un perrito, el cual necesita comida, pero ésta se encuentra fuera de la caja.

¿Que ocurriría en la vida real? El perro moriría. ¿Que pasa aquí? La caja que contiene el perro y necesita de la comida para el “buen funcionamiento” del perro mantiene y no pierde nunca la referencia de la comida, por lo que cuando el perro la necesita la puede obtener.

Javascript MVC

JMVC presenta una serie de herramientas para facilitar el desarrollo de aplicaciones web, centrándose principálmente en las aplicaciones RIA (Rich Internet Application). Como ellos mismos se enuncian, JMVC es un framework open-source que contiene las mejores ideas para el desarrollo jQuery.

En concreto, JMVC extiende jQuery y promueve una serie de mejores practicas que facilitan el desarrollo y mantenimiento del código, apoyándose para esto en el concepto CoC (convention over configuration).

Las áreas en la que se puede dividir JMVC son:

  1. CoC, este es un punto común en la mayoría de los framework modernos. JMVC establece una serie de ideas para organizar los proyectos. Entre las convenciones cabe destacar la organización de directorios y nomenclaturas.
  2. Testing, para poder realizar tantos test unitarios y funcionales como test de integración, presentando herramientas tanto para probar el código, validar los resultados, mostrar informes de los test, simular el servidor, etc.
  3. Documentación. Siguiendo estos estándares y junto a las herramientas de documentación que el framework aporta, se podrán generar informes con la documentación del proyecto. Similar a Javadoc.
  4. Dependencias. Este es un punto importante cuando los proyectos son de gran tamaño. JMVC usa la librería StealJS para gestionar las dependencias, y de nuevo, se apoya en sus recomendaciones de directorios y organizativas para que el uso y gestión de las dependencias sea algo sencillo.
  5. MVC. El framework extiende jQuery y crea una serie de herramientas que permiten la creación de la aplicación cumpliendo el patrón MVC. Además, aporta una serie de herramientas comunes para la mayoría de las aplicaciones.

Respecto al servidor, JMVC es independiente del servidor que expone los servicios. Y, como veremos en detalle en el modelo, JMVC podrá acceder de forma sencilla a cualquier servicio RESTful HTTP.

Estructura de directorios estándar

La forma de obtener buenos resultados en los proyectos grandes esta diréctamente relacionado con la forma en que se divide el trabajo, y por tanto, el proyecto se convierte en la unión de pequeños proyectos. Con este principio, JMVC propone una estructura de directorios que separa por completo cada modulo del proyecto.

Tradicionalmente en los proyecto jQuery, la estructura suele ser sencilla del tipo scripts, css, images, etc. Esta estructura se vuelve ineficiente cuando la aplicación crece, frecuentemente teniendo una enorme cantidad de plugins jQuery.

Para solucionar este problema JMVC divide cada recurso en su propio directorio.

\tabs

  tabs.js        - código para el widget tabs

  tabs.html      - pagina demostración del widget

  funcunit.html  - pagina para lanzar test

  tabs_test.js   - código de los test

  tabs.css       - css para el widget tabs

De esta forma, se permite trabajar en tabs de forma independiente al resto del sistema.

Una aplicación esta formada por diferentes recursos. JMVC divide estos recursos en tres tipos, modelos, controladores y plugins. Y estos recursos pueden agruparse en dos directorios, el directorio de aplicación y el de librería. Aunque este nivel de separación puede variar entre diferente proyectos. El directorio librería contendrá aquellos recursos que podrán reusarse en diferentes puntos de la aplicación o compartirse en otros proyectos. En cambio, el directorio aplicación contendrá aquellos recursos específicos de la aplicación.

A continuación se muestra una posible estructura de directorio para una aplicación de la empresa jupiter que quiere desarrollar la aplicación cms.

\cms

  \models    - models for the CMS

  \views     - views to configure the grid

  cms.js

\jupiter

  \tabs      - a basic tabs widget

  \edit      - binds a form to edit a model instance

  \grid      - a configurable grid

    \views

Modelo – Vista – Controlador

Definir una clara separación en capas del diseño una aplicación es siempre un tema complejo cuando se quiere construir una aplicación de fácil mantenimiento. JavascriptMVC aporta una división en tres capas.

  1. Ajax / Domain Model - Modelo
  2. Client Side Templates - Vista
  3. Eventos / Plugin – Controlador

Modelo

Los modelos propuestos por JMVC encapsulan las peticiones Ajax y encapsula los métodos relacionados con la gestión de estas peticiones. La forma más sencilla de crear un modelo es la siguiente.

$.Model('Todo', {});

Siguiendo las convenciones, JMVC presupone un servicio REST en su forma más estándar. En el siguiente ejemplo se ve como se modificar algunos de estos valores estándar y que además coinciden  con los valores que JMVC establece por defecto.

$.Model('Todo',{
 findAll: 'GET /todos.json',
 findOne: 'GET /todos/{id}.json',
 create:  'POST /todos.json',
 update:  'PUT /todos/{id}.json',
 destroy: 'DELETE /todos/{id}.json'
},{});

Crear una nueva instancia en el servidor.

// create a todo instance
var todo = new Todo({name: "do the dishes"})
// save it on the server
todo.save();

Obtener las instancias del servidor.

Todo.findAll({}, function( todos ){
 // print out the todo names
 $.each(todos, function(i, todo){
   console.log( todo.name );
 });
});

Actualizar una instancia en el servidor.

// update the todos' name
todo.attr('name','Take out the trash')
// update it on the server
todo.save()
Borrar una instancia en el servidor.
todo.destroy()

La información detallada del uso y las características de $.Model se pude ver en http://javascriptmvc.com/docs.html#!jQuery.Model.

Controlador

Los $.Controller son unas factorías de plugin jQuery que define muchas de las funcionalidades comunes en la mayoría de los plugins y añade nuevas funcionalidades como la herencia, variables de instancia y clase y gestión de eventos.

Ejemplo básico de un controlador para gestionar tabs.

$.Controller('Tabs',{
 click: function() {...},
 '.tab click' : function() {...},
 '.delete click' : function() {...}
})
$('#tabs').tabs();

Este controlador define un widget llamado tabs, que tiene una función ‘click’ que de forma automática es llamada cuando se hace click sobre el elemento donde se crea el controlador, y otros dos elementos que están asociados a los DOM con clases especificas.

Lo más importante a destacar es como permite definir recursos de forma estándar. Estos se pueden instanciar de la misma forma que se instancia los plugin jQuery. A la vez permite gestionar los eventos y asociarlos a funciones de forma sencilla y destacando que estas asociaciones serán eliminadas cuando se destruye el widget. Se explicará esto en el apartado de eventos.

Más información sobre los $.Controller en http://javascriptmvc.com/docs.html#!jQuery.Controller.

Vista

Las vistas en JMVC facilitan y convierten en algo práctico el uso de las plantillas en el lado del cliente. Permiten la definición de estas plantillas de forma independiente, comprimen estas plantillas y las empaquetan precompiladas para producción. Además, estas funcionalidades están integradas tanto para empaquetar como para usarlas e incluirlas en el DOM. En el siguiente ejemplo se muestra como representar una de estas plantillas.

$("#foo").html('//path/to/template.ejs', data);

Esta línea generara una vista a partir de template.ejs con los datos existentes en data, y la agregará al elemento foo.

JMVC viene con varios gestores de plantillas, aunque por defecto solo carga el motor EJS. El siguiente código muestra una plantilla EJS.

<ul>
<% for(var i=0; i<supplies.length; i++) {%>
  <li><%= supplies[i] %></li>
<% } %>
</ul> 

Integración de las vistas

A continuación se muestra un ejemplo de como se integran las tres capas, de forma que que generar una vista a partir de un modelo es sencillo y la funcionalidad queda claramente separada en las tres capas.

init : function(){  this.element.html(this.view()); }

Esta línea forma parte de un controlador dentro de la parte estáticas. El método init es llamado al crearse el controlador y this hace referencia a la instancia misma del controlador. Element es el elemento donde se creo el controlador y por ultimo this.view() representa la plantilla y la devuelve como cadena, por tanto al asociar dicha cadena al elemento se mostrará en la web. El método view no se le especifica que plantilla usar, pero JMVC establece que view buscará la plantilla de nombre init en el directorio views.

Testing, documentación y librería

JMVC viene con un motor de test que esta integrado con el resto de módulos. Viene provisto de plantillas de forma que al crear los recursos se crean automáticamente los elementos necesarios para la ejecución y validación de estos test.

Igualmente, viene provisto de un motor de documentación. Este analizará el código siguiendo las dependencias establecidas, y generará la documentación necesaria.

Por ultimo, JMVC dispone de una librería de plugins con las herramientas comunes y necesarias en la mayoría de los proyectos. Estas librerías están probadas y son de fácil integración, estos recursos permiten centrar los proyectos en la codificación en alto nivel y por tanto mejoran la calidad de este y reducen el tiempo de desarrollo.

Resumen JavaScriptMVC

JMVC permite realizar las tareas de gestión de código JS de la forma más sencilla posible. Entre los aspectos más destacados están

  1. Testing (especialmente test funcionales y automatización)
  2. Documentación
  3. Dividir el código en partes logícas y ordenadas
  4. Comprimir y concater código JS y CSS
  5. Usar y organizar plantillas del lado del cliente
  6. Crear plugins jQuery bien organizados y sin problemas de memoría
  7. Gestion de errores y logging

JMVC es un framewrok completo que intenta convertir jQuery en una plataforma de desarrollo para aplicaciónes de calidad empresarial. Complementa todas las deficiencias de este y aporta una base de desarrollo  con librerías y utilidades que permiten un inicio rápido de un proyecto.

Conclusiones

Todas las tecnologías expuestas son tecnologías de alto nivel. Tanto jQuery como JMVC requieren de experiencia para entender y dominar todas sus herramientas. En especial, el framework expuesto es un framework muy amplio y por tanto conocer y dominar todas sus partes es complejo.

Podemos concluir que
JavaScriptMVC parece un herramienta idónea para la organización y gestion de
proyectos complejos JavaScript, pero en cambio no parece ser la mejor opción
cuando se trata de proyectos pequeños.

Por último, REST presenta una arquitectura que facilita el desarrollo y acceso casi estándar a los recursos de un servidor. Actualmente es fácil encontrar servidores que implementen servicios RESTfull, y por tanto el acceso al servidor desde JMVC es sencillo e incluso trivial en muchos casos.
